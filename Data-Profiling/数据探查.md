# 数据探查 Skill（Data Profiling）

## 定位
在数仓开发启动前，基于源系统表结构和原始数据，系统性地探查数据现状，
输出可直接指导 DWD/DIM/DWS 三层建设的四类文档。

## 触发条件
以下任意一条成立时使用本 Skill：
- 拿到源系统表结构或数据库访问权限，准备开始数仓开发
- 需要评估源数据质量是否支撑指标计算
- 需要为业务访谈准备提纲
- 怀疑某张表存在数据质量问题影响指标准确性

---

## 输入规格

### 必须具备
| 输入项 | 说明 | 最低要求 |
|--------|------|---------|
| 表结构（DDL） | 所有源系统表的建表语句或字段清单 | 字段名 + 字段类型 + 注释（有则最好） |
| 原始数据访问 | 能对源系统数据执行 SELECT 查询 | 至少有读权限，数据量不限 |

### 有则更好
| 输入项 | 说明 | 能补充什么 |
|--------|------|-----------|
| 数据字典 | 字段业务含义说明文档 | 减少字段含义猜测，提高探查准确性 |
| 历史手工报表 | 业务方之前手工维护的 Excel | 提供指标口径参考 |
| 系统操作手册 | 源系统的用户手册或功能说明 | 帮助理解枚举值和业务规则 |

---

## 处理过程（6个步骤）

---

### Step 1：表结构分析
**目标**：从表名和字段名推断业务域划分和核心实体

#### 1.1 表名聚类 → 识别业务域
```
操作：将所有表名按前缀/关键词分组

示例：
t_policy_*    → 保单域
t_claim_*     → 理赔域
t_cust_*      → 客户域
t_agent_*     → 销售域
t_finance_*   → 财务域
t_product_*   → 产品域
t_org_*       → 机构域
```

**输出**：业务域初步划分清单（草稿）

#### 1.2 高频字段统计 → 发现核心实体
```sql
-- 统计各字段名在所有表中出现的次数（需遍历 information_schema）
SELECT column_name, COUNT(*) AS appear_cnt
FROM information_schema.columns
WHERE table_schema = '目标库'
GROUP BY column_name
ORDER BY appear_cnt DESC
LIMIT 30;
```

**判断规则**：
- 出现 > 20 张表 → 核心关联键，必须建 DIM 维度表
- 出现 10-20 张表 → 重要业务字段，需重点关注质量
- 出现 < 5 张表 → 域内私有字段

#### 1.3 时间字段组合分析 → 还原业务流程节点
```
操作：找同一业务实体下的所有 *_date / *_time 字段

示例（保单表）：
apply_time    申请时间
submit_time   提交时间
audit_time    核保时间
effect_time   生效时间
renew_time    续保时间
lapse_time    失效时间
surrender_time 退保时间

→ 推断业务流程：申请 → 提交 → 核保 → 生效 → (续保/失效/退保)
```

#### 1.4 识别事实表 vs 维度表
```
事实表特征（满足 3 条以上）：
  ✓ 有业务发生时间字段
  ✓ 有金额/数量等度量字段
  ✓ 有大量外键（关联多个实体）
  ✓ 数据量大（通过 COUNT 粗估）
  ✓ 每天持续增长

维度表特征（满足 3 条以上）：
  ✓ 有 name / desc / label 类字段
  ✓ 有层级字段（parent_code / level_code）
  ✓ 有生效/失效时间（暗示需要拉链表）
  ✓ 数据量小且相对稳定
  ✓ 被多张事实表关联
```

**⚠️ 注意**：发现有 `effective_date` + `expire_date` + `is_current` 组合的表，
记录为"需要拉链表"，后续 DIM 层建设重点处理。

### Step 2：完整性探查
**目标**：评估核心字段的可用性，发现数据断档

#### 2.1 核心字段空值率
```sql
SELECT
    COUNT(*) AS total,
    -- 主键
    ROUND(SUM(CASE WHEN policy_no IS NULL THEN 1 ELSE 0 END)*100.0/COUNT(*),2) AS policy_no_null_pct,
    -- 业务时间
    ROUND(SUM(CASE WHEN effect_date IS NULL THEN 1 ELSE 0 END)*100.0/COUNT(*),2) AS effect_date_null_pct,
    -- 关键维度
    ROUND(SUM(CASE WHEN org_code IS NULL THEN 1 ELSE 0 END)*100.0/COUNT(*),2) AS org_code_null_pct,
    ROUND(SUM(CASE WHEN agent_code IS NULL THEN 1 ELSE 0 END)*100.0/COUNT(*),2) AS agent_code_null_pct,
    -- 度量字段
    ROUND(SUM(CASE WHEN prem_amt IS NULL THEN 1 ELSE 0 END)*100.0/COUNT(*),2) AS prem_amt_null_pct
FROM t_policy_main;
```

**判断标准**：
| 空值率 | 字段可用性 | 处理建议 |
|--------|-----------|---------|
| = 0% | 可直接作为关联键 | 无需处理 |
| 0-1% | 基本可用 | DWD 层过滤空值并记录 |
| 1-10% | 有风险 | 分析原因，确认是否影响指标 |
| > 10% | 不可直接使用 | 必须访谈业务方，确认处理方式 |

#### 2.2 数据断档检测
```sql
-- 检查最近 90 天各分区是否有缺失
SELECT
    dt,
    COUNT(*) AS row_cnt,
    SUM(prem_amt) AS total_prem
FROM t_policy_main
WHERE dt >= DATE_SUB(CURRENT_DATE, 90)
GROUP BY dt
ORDER BY dt;

-- 若发现某天 row_cnt = 0 或较前后天差异 > 50%，标记为断档风险
```

#### 2.3 历史可追溯范围
```sql
SELECT
    MIN(dt) AS earliest_dt,
    MAX(dt) AS latest_dt,
    COUNT(DISTINCT dt) AS available_days
FROM t_policy_main;
```

### Step 3：唯一性 & 一致性探查
**目标**：确认关联键可靠性，发现跨系统数据孤岛

#### 3.1 主键唯一性
```sql
-- 检测重复主键
SELECT policy_no, COUNT(*) AS dup_cnt
FROM t_policy_main
GROUP BY policy_no
HAVING COUNT(*) > 1
ORDER BY dup_cnt DESC
LIMIT 50;

-- 汇总重复率
SELECT
    COUNT(*) AS total,
    COUNT(DISTINCT policy_no) AS unique_cnt,
    ROUND((COUNT(*) - COUNT(DISTINCT policy_no))*100.0/COUNT(*),2) AS dup_rate
FROM t_policy_main;
```

**若发现重复**，进一步查重复原因：
```sql
-- 查看重复的 policy_no 具体有哪些差异字段
SELECT *
FROM t_policy_main
WHERE policy_no IN (
    SELECT policy_no FROM t_policy_main GROUP BY policy_no HAVING COUNT(*) > 1
)
ORDER BY policy_no, create_time;
-- 观察：重复行的差异在哪些字段？是批改导致？还是系统 bug？
```

#### 3.2 跨系统编码一致性
```sql
-- 机构编码在两个系统是否一致
SELECT '仅在核心系统' AS source, org_code FROM t_policy_main
WHERE org_code NOT IN (SELECT org_code FROM t_org)
UNION ALL
SELECT '仅在机构表' AS source, org_code FROM t_org
WHERE org_code NOT IN (SELECT DISTINCT org_code FROM t_policy_main);

-- 代理人编码在两个系统是否一致（CRM vs 核心）
SELECT COUNT(DISTINCT a.agent_code) AS core_only
FROM t_policy_main a
LEFT JOIN t_agent_crm b ON a.agent_code = b.agent_code
WHERE b.agent_code IS NULL;
```

#### 3.3 枚举值分布
```sql
-- 状态字段的所有枚举值及占比
SELECT
    policy_status,
    COUNT(*) AS cnt,
    ROUND(COUNT(*)*100.0/SUM(COUNT(*)) OVER(),2) AS pct
FROM t_policy_main
GROUP BY policy_status
ORDER BY cnt DESC;

-- ⚠️ 重点关注：
-- 1. 不认识的枚举值（如 'XB', 'ZZ'）→ 加入访谈提纲
-- 2. NULL 值 → 按空值率判断标准处理
-- 3. 空字符串 '' → 比 NULL 更隐蔽，单独检查
SELECT COUNT(*) FROM t_policy_main WHERE policy_status = '';
```

### Step 4：准确性 & 时效性探查
**目标**：发现异常值和数据波动，评估调度时序

#### 4.1 数值字段分布
```sql
SELECT
    MIN(prem_amt)                        AS min_val,
    MAX(prem_amt)                        AS max_val,
    AVG(prem_amt)                        AS avg_val,
    PERCENTILE(prem_amt, 0.01)           AS p01,
    PERCENTILE(prem_amt, 0.25)           AS p25,
    PERCENTILE(prem_amt, 0.50)           AS p50,
    PERCENTILE(prem_amt, 0.75)           AS p75,
    PERCENTILE(prem_amt, 0.99)           AS p99,
    SUM(CASE WHEN prem_amt < 0 THEN 1 ELSE 0 END) AS negative_cnt,
    SUM(CASE WHEN prem_amt = 0 THEN 1 ELSE 0 END) AS zero_cnt
FROM t_policy_main;
```

#### 4.2 时间逻辑校验
```sql
-- 出险日不能早于生效日
SELECT COUNT(*) AS err_cnt
FROM t_claim_report
WHERE accident_date < effect_date;

-- 退保日不能早于生效日
SELECT COUNT(*) AS err_cnt
FROM t_policy_surrender
WHERE surrender_date < effect_date;

-- 核保日不能早于申请日
SELECT COUNT(*) AS err_cnt
FROM t_policy_main
WHERE audit_time < apply_time;
```

#### 4.3 数据量趋势与波动
```sql
-- 近 30 天数据量 + 环比波动
SELECT
    dt,
    row_cnt,
    LAG(row_cnt) OVER(ORDER BY dt)  AS prev_cnt,
    ROUND((row_cnt - LAG(row_cnt) OVER(ORDER BY dt))*100.0/
          NULLIF(LAG(row_cnt) OVER(ORDER BY dt),0), 2) AS wow_pct
FROM (
    SELECT dt, COUNT(*) AS row_cnt
    FROM t_policy_main
    WHERE dt >= DATE_SUB(CURRENT_DATE, 30)
    GROUP BY dt
) t
ORDER BY dt;

-- ⚠️ 波动超过 ±30% 的日期需要单独分析原因
```

#### 4.4 数据到达时效
```sql
-- 评估数据从业务发生到落库的延迟
SELECT
    AVG(DATEDIFF(create_time, effect_date)) AS avg_lag_days,
    MAX(DATEDIFF(create_time, effect_date)) AS max_lag_days,
    SUM(CASE WHEN DATEDIFF(create_time, effect_date) > 1 THEN 1 ELSE 0 END) AS t2_cnt
FROM t_policy_main
WHERE dt = DATE_SUB(CURRENT_DATE, 1);
-- 结果决定 DWS 层调度时间窗口（T+1 还是 T+2 才可靠）
```

### Step 5：关联性探查
**目标**：评估跨表 JOIN 可靠性，发现数据孤岛

#### 5.1 核心业务路径追踪
```sql
-- 以 policy_no 为线索，追踪它在各表的出现情况
SELECT '申请表' AS tbl, COUNT(DISTINCT policy_no) AS cnt FROM t_policy_apply
UNION ALL
SELECT '承保表',  COUNT(DISTINCT policy_no) FROM t_policy_main
UNION ALL
SELECT '批改表',  COUNT(DISTINCT policy_no) FROM t_policy_endorse
UNION ALL
SELECT '续保表',  COUNT(DISTINCT policy_no) FROM t_policy_renew
UNION ALL
SELECT '理赔表',  COUNT(DISTINCT policy_no) FROM t_claim_report
UNION ALL
SELECT '退保表',  COUNT(DISTINCT policy_no) FROM t_policy_surrender;

-- 各表数量的比例关系 → 还原业务主路径
```

#### 5.2 跨表 JOIN 匹配率
```sql
-- 理赔单能关联上保单的比例
SELECT
    COUNT(*)                                              AS total_claims,
    SUM(CASE WHEN p.policy_no IS NOT NULL THEN 1 ELSE 0 END) AS matched,
    ROUND(SUM(CASE WHEN p.policy_no IS NOT NULL THEN 1 ELSE 0 END)*100.0/COUNT(*),2) AS match_rate
FROM t_claim_report c
LEFT JOIN t_policy_main p ON c.policy_no = p.policy_no;

-- 匹配率判断标准
-- ≥ 99%：可直接 JOIN，无需特殊处理
-- 95-99%：可用，DWM 层需处理孤儿记录
-- < 95%：高风险，必须访谈，查明原因
```

#### 5.3 维度表覆盖率
```sql
-- 事实表中的机构编码在维度表中的覆盖情况
SELECT
    COUNT(DISTINCT f.org_code)  AS fact_org_cnt,
    COUNT(DISTINCT d.org_code)  AS matched_org_cnt,
    COUNT(DISTINCT CASE WHEN d.org_code IS NULL THEN f.org_code END) AS missing_cnt,
    ROUND(COUNT(DISTINCT d.org_code)*100.0/COUNT(DISTINCT f.org_code),2) AS coverage_rate
FROM t_policy_main f
LEFT JOIN t_org d ON f.org_code = d.org_code;
```

### Step 6：业务规律挖掘
**目标**：从数据中推断隐性业务规则，丰富建模认知

#### 6.1 业务状态机还原
```sql
-- 如有操作日志表，还原状态转换路径
SELECT
    from_status,
    to_status,
    COUNT(*) AS trans_cnt,
    ROUND(COUNT(*)*100.0/SUM(COUNT(*)) OVER(PARTITION BY from_status),2) AS trans_rate
FROM t_policy_status_log
GROUP BY from_status, to_status
ORDER BY from_status, trans_cnt DESC;

-- 高频路径 = 主流程，低频路径 = 异常/特殊流程
```

#### 6.2 业务周期识别
```sql
-- 月内日期分布 → 发现月末冲量等业务周期
SELECT
    DAY(effect_date) AS day_of_month,
    COUNT(*) AS cnt
FROM t_policy_main
GROUP BY DAY(effect_date)
ORDER BY day_of_month;

-- 小时分布 → 判断是否有线上渠道（非纯线下柜面）
SELECT
    HOUR(create_time) AS hour,
    COUNT(*) AS cnt
FROM t_policy_apply
GROUP BY HOUR(create_time)
ORDER BY hour;
```

#### 6.3 隐性规则推断
```sql
-- 保费金额分布 → 判断是标准化产品还是浮动费率
SELECT
    ROUND(prem_amt, -2) AS prem_bucket,
    COUNT(*) AS cnt
FROM t_policy_main
GROUP BY ROUND(prem_amt, -2)
ORDER BY cnt DESC
LIMIT 20;

-- 单一机构的件量占比 → 发现是否有高度集中风险
SELECT
    org_code,
    COUNT(*) AS cnt,
    ROUND(COUNT(*)*100.0/SUM(COUNT(*)) OVER(),2) AS pct
FROM t_policy_main
GROUP BY org_code
ORDER BY cnt DESC
LIMIT 10;
```

## 探查注意事项

### 通用原则
- ⚠️ 空字符串 `''` 和 `NULL` 都要检查，两者都代表数据缺失
- ⚠️ 先查数据量（COUNT）再做分布探查，避免全表扫描超时
- ⚠️ 探查结论要标注置信度，不能把推断当确认
- ⚠️ 每个质量问题必须标注"影响哪个指标"，否则无法判断严重程度

### 保险行业特殊关注点
- 必须探查所有时间字段（承保日/交费日/会计日/出险日），口径不同指标结果差异显著
- 再保数据和原保数据可能在不同表，探查时注意是否有分出/摊回的标识字段
- 代理人和机构表必须检查历史变更字段，这两个维度几乎必须建拉链表

### 探查质量自查
完成全部6个步骤后，能回答以下问题则探查合格：

- [ ] 数仓域划分是否已确定？
- [ ] 核心关联键是否已确认（空值率=0、唯一性达标）？
- [ ] DIM层哪些表需要拉链是否已判断？
- [ ] 影响指标计算的数据质量问题是否都已识别？
- [ ] 访谈提纲是否已生成（至少10个有数据依据的问题）？
- [ ] DWS调度时序是否已有初步结论？
