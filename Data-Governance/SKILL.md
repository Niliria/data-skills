# 数据治理 · DDL/SQL/血缘规范检查

## 概述

本 skill 是数据治理规范检查的 **主入口**，根据用户的具体诉求自动路由到对应的子 skill。
如果用户没有明确指定检查维度，默认依次执行全部四个子 skill，最终汇总结果。

### 子 skill 清单

| 子 skill | 文件 | 职责 |
|----------|------|------|
| 表级 DDL 规范检查 | [SKILL-table.md](SKILL-table.md) | 检查表命名、分层前缀、字符规范、语义清晰度、表注释 |
| 字段级 DDL 规范检查 | [SKILL-field.md](SKILL-field.md) | 检查字段命名、字段类型、字段注释、字段设计合理性 |
| SQL 代码规范检查 | [SKILL-sql.md](SKILL-sql.md) | 检查 SQL 安全合规、性能优化、代码规范性 |
| 血缘健康检查 | [SKILL-lineage.md](SKILL-lineage.md) | 检查表间依赖关系、越层依赖、血缘覆盖率、链路复杂度 |

---

## 路由规则

收到用户输入后，按以下优先级判断需要执行哪些子 skill：

### 仅执行表级检查（SKILL-table.md）

用户明确提到以下关键词之一：
- "表命名"、"表注释"、"分层前缀"、"表级规范"、"表级审查"、"表结构检查"

### 仅执行字段级检查（SKILL-field.md）

用户明确提到以下关键词之一：
- "字段规范"、"字段命名"、"字段类型"、"字段注释"、"字段设计"、"字段问题"、"字段审查"

### 仅执行 SQL 检查（SKILL-sql.md）

用户明确提到以下关键词之一：
- "SQL检查"、"代码审查"、"SQL规范"、"代码规范检查"、"SQL性能"、"安全合规检查"

### 仅执行血缘检查（SKILL-lineage.md）

用户明确提到以下关键词之一：
- "血缘检查"、"血缘分析"、"依赖检查"、"越层依赖"、"数据链路"、"血缘健康"、"表依赖"

### 执行全部子 skill（默认行为）

以下情况按顺序执行四个子 skill：
- 用户没有明确指定检查维度（如"帮我检查下这个 DDL"、"审查一下"）
- 用户提到宽泛的关键词（如"数据治理"、"数仓规范"、"DDL 审查"、"建表语句检查"）
- 用户同时提到多个维度的诉求

## 执行流程

### Step 1：解析输入，按内容拆分

用户可能输入以下一种或多种内容：
- **DDL 内容**：包含 `CREATE TABLE` 的建表语句（文件上传或直接粘贴）
- **SQL 内容**：包含 `SELECT`、`INSERT`、`UPDATE`、`DELETE`、`MERGE` 的查询/加工语句（文件上传或直接粘贴）

从输入中提取并拆分：
- 所有 `CREATE TABLE` 语句 → 归为 DDL 内容
- 所有查询/加工语句 → 归为 SQL 内容
- 如果用户只上传了一个文件，按语句类型自动拆分

### Step 2：判断路由

根据用户输入中的关键词，按上述路由规则确定需要执行的子 skill。

### Step 3：输入匹配检查

对每个需要执行的子 skill，检查是否有对应类型的输入：
- 表级检查 + 字段级检查：需要 DDL 内容
- SQL 检查：需要 SQL 内容
- 血缘检查：需要 SQL 内容（含 INSERT INTO / SELECT 的语句）

如果某子 skill 缺少对应类型的输入，跳过该 skill 并在报告中注明"跳过原因"。

### Step 4：执行子 skill

对每个有对应输入的子 skill，依次读取其文件，按照其定义的规则库和工作流执行检查。
每个子 skill 独立执行，互不影响。

执行顺序：表级 → 字段级 → SQL → 血缘。

### Step 5：汇总结果（仅全量执行时）

当执行了两个及以上子 skill 时，生成一份综合报告：


### Step 6：输出

单个子 skill 执行时，按该子 skill 定义的格式输出 Excel。
全量执行时，输出一个综合 Excel 文件，包含：
- 「表级问题」sheet（来自表级检查）
- 「字段级问题」sheet（来自字段检查）
- 「代码问题」sheet（来自 SQL 检查）
- 「血缘问题」sheet（来自血缘检查）
- 「综合摘要」sheet（汇总统计，含各专项执行状态）
