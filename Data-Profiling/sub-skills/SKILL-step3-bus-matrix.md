## 执行步骤

### Step 1: 维度识别

**目标：** 系统化识别所有候选维度。

**操作方法：5W1H 框架**

对每个业务过程（事实表候选），从以下角度识别维度：

| 维度角度 | 问题 | 分析方法 |
|---------|------|---------|
| Who（谁） | 谁参与了该业务过程？ | 查找客户ID、代理人ID等外键字段 |
| What（什么） | 涉及什么产品/服务？ | 查找产品ID、险种ID等外键字段 |
| When（何时） | 何时发生？ | 查找时间戳字段 → 时间维度 |
| Where（哪里） | 在哪里发生？ | 查找机构ID、地域编码等字段 |
| How（怎样） | 通过什么方式？ | 查找渠道ID、支付方式等字段 |
| Why（为什么） | 原因是什么？ | 查找原因编码等字段 |

**操作：**

1. **外键扫描**：遍历所有事实表候选，提取外键字段

2. **维度聚类**：相同业务含义的外键归为同一维度

3. **维度定义**：为每个维度明确名称、核心字段、数据来源、SCD 处理建议

**SCD 类型选择规则：**

| SCD 类型 | 适用场景 | 示例 |
|---------|---------|------|
| Type 1（覆盖） | 维度属性变化不影响历史分析 | 客户联系方式变更 |
| Type 2（新增版本） | 维度属性变化需要保留历史 | 客户地址变更、代理人职级变更 |
| Type 3（新增列） | 只需保留最近一次变更 | 机构名称变更（保留原名） |

### Step 2: 总线矩阵构建

**目标：** 构建 Kimball 标准总线矩阵。

**操作：**

1. **纵轴**：列出所有数据域和业务过程（来自Step 2 产出）
2. **横轴**：列出所有候选维度（来自 Step 1 产出）
3. **填充矩阵**：逐行分析每个业务过程涉及哪些维度
   - 判断依据：事实表中是否有对应维度的外键字段
   - 采样数据验证：确认外键关联是否真实存在
4. **补充度量列**：在矩阵右侧列出每个业务过程的关键度量

## 总线矩阵校验规则

| 校验项 | 说明 | 处理方式 |
|-------|------|---------|
| 空行检查 | 某业务过程所有维度都是 N | 检查是否遗漏维度 |
| 空列检查 | 某维度所有业务过程都是 N | 检查维度是否有意义 |
| 全 Y 列检查 | 某维度所有业务过程都是 Y | 正常，通常为时间维度 |
| 一致性检查 | 同一数据域内维度差异过大 | 检查数据域划分是否合理 |
| 完整性检查 | 总线矩阵是否覆盖了所有业务过程 | 补充遗漏的过程 |

## 注意事项

1. **维度粒度**：维度不宜过多（一般 8-15 个），过细会导致矩阵稀疏
2. **命名统一**：维度命名与数据域、业务过程命名保持一致的命名规范
3. **度量标注**：每个业务过程至少标注 1 个关键度量
4. **不要过度推断**：如果无法从表结构确认某个维度关联，标记 `❓` 而非猜测
5. **一致性维度是核心**：一致性维度是数仓建设的基石，务必准确标注
