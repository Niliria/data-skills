### 1.2 系统配置表过滤（不接入数仓）

```sql
SELECT table_schema, table_name
FROM information_schema.tables

  AND (table_name LIKE 't_sys_%'
    OR table_name LIKE 't_log_%'
    OR table_name LIKE 't_dict_%'
    OR table_name LIKE 't_config_%'
    OR table_name LIKE 't_param_%'
    OR table_name LIKE 't_menu_%'
    OR table_name LIKE 't_role_%'
    OR table_name LIKE 't_permission_%');
```

**探查结果：**

| `t_sys_*` | t_sys_config、t_sys_param | 8张 | 系统运行参数，无业务分析价值 |

**✅ 过滤结论：**

## 二、基于 DDL 的探查（6 项）

仅凭表名和字段结构（DDL），无需查询任何数据，就能得出以下 6 项结论。

### 2.1 表名聚类 → 域划分

**探查操作：** 对 48 张业务表按表名关键词分组

```sql
SELECT
    CASE
        WHEN table_name LIKE 'policy%'  THEN '保单域'
        WHEN table_name LIKE 'claim%'   THEN '理赔域'
        WHEN table_name LIKE 'customer%'THEN '客户域'
        WHEN table_name LIKE 'agent%'   THEN '代理人域'
        WHEN table_name LIKE 'premium%' THEN '收付域'
        WHEN table_name LIKE 'org%'     THEN '机构域'
        WHEN table_name LIKE 'product%' THEN '产品域'
        ELSE '其他'
    END AS domain,
    COUNT(*) AS table_cnt
FROM information_schema.tables
WHERE table_schema IN ('db_policy','db_claim','db_customer','db_agent','db_finance','db_config')
GROUP BY 1 ORDER BY 2 DESC;
```

**探查结果：**

```
域名        表数量
保单域        18
理赔域        14
收付域        12
代理人域      10
客户域         8
机构域         3
产品域         3
```

**✅ 探查发现 1：域划分一次成型**

> 直接得出数仓 7 个业务域，无需任何文档。
> 数仓 DWD 层按此划分建子库：
> `dwd_policy / dwd_claim / dwd_premium / dwd_agent / dwd_customer / dwd_dim`

### 2.2 高频字段统计 → 发现核心实体

**探查操作：** 扫描所有字段名出现频次，找出跨表高频字段

```sql
SELECT column_name, COUNT(DISTINCT table_name) AS appear_in_tables
FROM information_schema.columns
WHERE table_schema IN ('db_policy','db_claim','db_customer','db_agent','db_finance','db_config')
GROUP BY column_name
HAVING COUNT(DISTINCT table_name) >= 3
ORDER BY appear_in_tables DESC;
```

**探查结果：**

```
字段名            出现表数   判断
policy_no           18     核心主键，保单是中心实体
org_code            15     最高频维度字段 → 必须建 dim_org
agent_code          12     高频维度 → 必须建 dim_agent
cust_id             10     核心实体 → 必须建 dim_customer
create_time         10     通用字段，非关联键
product_code         8     产品维度 → 建 dim_product
claim_no             7     理赔主键
channel_code         7     渠道维度 → 建 dim_channel
prem_amt             6     度量字段，非关联键
update_time          6     通用字段
```

**✅ 探查发现 2：核心实体与维度表来源**

> `policy_no` 出现 18 次，是贯穿整个系统的核心主键——保单是中心实体。
> `org_code / agent_code / cust_id / product_code / channel_code` 是 5 个高频维度字段，
> 直接对应 5 张 DIM 维度表，无需文档即可确定。

### 2.3 时间字段组合 → 还原业务流程节点

**探查操作：** 找核心事实表中所有时间类字段

```sql
SELECT column_name, column_comment
FROM information_schema.columns
WHERE table_name IN ('t_policy_main','t_policy_apply','t_claim_report','t_premium_recv')
  AND (column_name LIKE '%date%' OR column_name LIKE '%time%')
ORDER BY table_name, ordinal_position;
```

**探查结果：**

```
t_policy_apply:   apply_date（申请日期）、create_time、update_time
t_policy_main:    effect_date（生效日期）、expire_date（到期日期）、
                  underwrite_date（承保日期）、issue_date（出单日期）、
                  create_time、update_time
t_claim_report:   accident_date（出险日期）、report_date（报案日期）、
                  create_time、update_time
t_premium_recv:   recv_date（到账日期）、create_time
t_policy_status_log: change_time（变更时间）
```

**✅ 探查发现 3：完整业务时间轴一次性梳理清楚**

```
申请日期(apply_date)
    ↓ 平均2天
承保日期(underwrite_date)
    ↓ 平均1天
出单日期(issue_date)
    ↓ 即时
生效日期(effect_date)
    ↓ 持续N年
    ├─ 出险日期(accident_date)
    │      ↓ 平均1.3天
    │  报案日期(report_date)
    ↓
到期日期(expire_date)
```

> **数仓建设指导：** DWD 层所有时间字段都要保留，不同指标用不同时间口径：
> - 新单保费 → 按 `effect_date`（生效口径）
> - 新单件数 → 按 `underwrite_date`（承保口径）
> - 财务收入 → 按 `recv_date`（到账口径）

### 2.4 区分事实表与维度表

仅凭字段出现频次还不足以划分表的类型，需要对每张表的字段组成做特征检测。

**判定规则：**

| 事实表特征（满足 3 条以上判定为事实表） | 维度表特征（满足 2 条以上判定为维度表） |
|---|---|
| ✓ 有业务发生时间字段 | ✓ 有 name / desc / label 类字段 |
| ✓ 有金额 / 数量等度量字段 | ✓ 有 parent_code / level_code 层级字段 |
| ✓ 外键字段数量 > 3 个 | ✓ 有 effective_date + expire_date 组合 |
| ✓ 有 create_time / update_time 字段 | ✓ 字段数量少，外键少 |

**探查操作：** 对 11 张核心表逐一做字段特征扫描

```sql
SELECT
    table_name,
    -- 事实表特征
    SUM(CASE WHEN column_name LIKE '%_time' OR column_name LIKE '%_date'
             OR column_name LIKE '%_dt'  THEN 1 ELSE 0 END) AS time_field_cnt,
    SUM(CASE WHEN column_name LIKE '%amt'  OR column_name LIKE '%_num'
             OR column_name LIKE 'prem%' OR column_name LIKE 'claim%'
             OR column_name LIKE '%count' THEN 1 ELSE 0 END) AS measure_field_cnt,
    SUM(CASE WHEN column_name IN
        ('cust_id','policy_no','agent_code','org_code','product_code',
         'channel_code','claim_no','apply_id') THEN 1 ELSE 0 END) AS fk_field_cnt,
    SUM(CASE WHEN column_name IN ('create_time','update_time') THEN 1 ELSE 0 END) AS audit_field_cnt,
    -- 维度表特征
    SUM(CASE WHEN column_name LIKE '%name' OR column_name LIKE '%desc'
             OR column_name LIKE '%label' THEN 1 ELSE 0 END) AS name_desc_cnt,
    SUM(CASE WHEN column_name LIKE '%parent%' OR column_name LIKE '%level%'
             THEN 1 ELSE 0 END) AS hierarchy_cnt,
    SUM(CASE WHEN column_name IN ('effective_date','expire_date','start_date','end_date')
             THEN 1 ELSE 0 END) AS scd_range_cnt,
    COUNT(*) AS total_fields
FROM information_schema.columns
WHERE table_schema IN ('db_policy','db_claim','db_customer','db_agent','db_finance','db_config')
GROUP BY table_name
ORDER BY table_name;
```

**探查结果：**

        WHEN column_name = 'policy_no'  THEN 't_policy_main'

### 4.4 访谈提纲

基于以上探查，以下问题必须在开发前通过业务访谈确认：

#### 🔴 必须确认（高优先级）

**Q1. policy_status = 'XB' 是什么含义？**
数据现象：125条保单状态为 'XB'，DDL注释中未定义此枚举值。
需确认：该状态是已退市的历史状态？还是在用的特殊业务状态（如信用卡保险批次）？
影响决策：是否计入有效件数、是否过滤。

**Q2. channel_code 空值 3.2% 的原因是什么？**
数据现象：4,000条保单无渠道编码，且集中在2019-2020年。
需确认：是旧系统未记录渠道，还是存在一种无渠道的直销模式？
影响决策：渠道维度缺失会导致渠道分析报表不全。

**Q3. 7.8% 的数据 T+2 才到达，是否可以接受 T+2 的报表时效？**
数据现象：平均数据延迟1.3天，7.8%的保单在T+2日才落入系统。
需确认：业务方能否接受日报数据为前天数据，还是必须要T+1？
影响决策：若必须T+1，需要与源系统团队沟通优化数据链路。

**Q4. agent_code 空值的1,065条是什么业务场景？**
数据现象：1,065条保单无代理人编码，占比0.85%。
需确认：是网销直投（无代理人）？还是公司直销？还是数据录入遗漏？
影响决策：代理人业绩指标是否将这部分计入特定渠道。

#### 🟡 建议确认（中优先级）

**Q5. prem_amt = 0 的10条保单是否为团体险豁免保费？**
数据现象：10条保单保费为0，保额正常。
推测：可能是团体险主险豁免附加险保费的场景。
影响决策：新单保费指标是否将这10条过滤。

**Q6. 新单件数的时间口径是承保日还是生效日？**
业务背景：两个口径数值差异在0.5%-2%之间。
需确认：管理层驾驶舱、监管报表、内部考核各用什么口径。
影响决策：DWS层需要同时保留两个时间字段的聚合结果。

**Q7. 历史12条理赔孤儿记录是否需要补录对应保单？**
数据现象：12条2019年理赔记录找不到对应保单（旧系统迁移遗留）。
需确认：这12条对应的历史保单是否有旧系统数据可以补录。
影响决策：影响2019年赔付率的历史数据准确性。

## 五、覆盖度分析：探查能解决什么 / 不能解决什么

### 80/20 法则

> 通过 12 项探查，可以解决 **80%** 的数仓建设问题。
> 剩余 **20%** 需要通过业务访谈和文档补充来解决。

| 探查能解决（80%） | 探查不能解决（20%） |
|---|---|
| ✓ 域划分（7个业务域） | ✗ 指标的业务定义（如"有效件"是否含批改中） |
| ✓ 核心实体识别（保单为中心） | ✗ 枚举值的业务含义（如 XB 状态） |
| ✓ 事实表/维度表划分（7+5） | ✗ 空值字段的业务原因（如 channel_code 缺失） |

| ✓ 数据质量问题定位（8个问题） | ✗ 跨系统数据一致性标准 |
