# 数仓业务建模 Skill

## 概述

本 Skill 从业务系统的 DDL 表结构和采样数据出发，通过 3 个步骤完成数仓业务建模：

1. **业务过程识别**：从表结构中识别业务板块、业务过程，绘制流程草图
2. **数据域划分**：将业务过程归纳为数据域
3. **总线矩阵构建**：定义一致性维度，构建 Kimball 标准总线矩阵

**产出**：3 个 Markdown 文档，每个步骤产出 1 个结果文档。

## 前置要求

### 必选输入

| 输入项 | 格式 | 说明 |
|-------|------|------|
| 表结构 DDL | SQL 文本 / .sql 文件 | 所有业务表的 CREATE TABLE 语句 |

### 推荐输入

| 输入项 | 格式 | 说明 |
|-------|------|------|
| 采样数据 | CSV / Markdown 表格 | 每表 10-20 条代表性数据，用于验证推断 |

## 输入案例

### 案例一：提供 DDL 文件（最简输入）

用户直接提供 `.sql` 文件或粘贴 DDL 文本：

> **用户说：** 帮我做数仓业务建模，这是我们的 DDL 文件

然后提供文件路径：
- examples/sample_ddl.sql
- /path/to/your/tables.sql

或者直接粘贴 DDL 文本：

```sql
CREATE TABLE t_policy (
  policy_id BIGINT COMMENT '保单ID，主键',
  policy_no VARCHAR(32) COMMENT '保单号',
  customer_id BIGINT COMMENT '客户ID，关联t_customer',
  product_id BIGINT COMMENT '产品ID，关联t_product',
  total_premium DECIMAL(18,2) COMMENT '总保费，单位：元',
  status VARCHAR(10) COMMENT '保单状态：1-有效 2-失效 3-退保',
  sign_date DATE COMMENT '签单日期',
  create_time TIMESTAMP COMMENT '创建时间'
) COMMENT '承保业务-保单主表';
```

**完整 DDL 示例**：参见 [examples/sample_ddl.sql](examples/sample_ddl.sql)（保险行业 20+ 张表）

## 输入格式详细规范

### DDL 格式

**支持的 DDL 方言**：MySQL、PostgreSQL、Oracle、Hive、Spark SQL

**必须包含的信息**：
- 表名
- 字段名、字段类型
- COMMENT 注释（表注释和字段注释，强烈建议提供）

**可选但推荐的信息**：
- 主键约束（PRIMARY KEY）
- 外键约束（FOREIGN KEY / REFERENCES）
- 分区定义（PARTITIONED BY）

**示例**：
```sql
-- 推荐：包含完整 COMMENT
CREATE TABLE t_policy (
  policy_id BIGINT COMMENT '保单ID，主键',
  customer_id BIGINT COMMENT '客户ID，关联t_customer',
  status VARCHAR(10) COMMENT '状态：1-有效 2-失效 3-退保',
  create_time TIMESTAMP COMMENT '创建时间'
) COMMENT '承保业务-保单主表';

-- 可用但置信度低：缺少 COMMENT
CREATE TABLE t_policy (
  policy_id BIGINT,
  customer_id BIGINT,
  status VARCHAR(10),
  create_time TIMESTAMP
);
```

### 采样数据格式

**推荐格式**：Markdown 表格（每表 10-20 条）

| policy_id | policy_no | customer_id | status | premium |
|-----------|-----------|-------------|--------|---------|
| 1001 | POL001 | 5001 | 1 | 12000.00 |
| 1002 | POL002 | 5002 | 1 | 3600.00 |
| 1003 | POL003 | 5003 | 3 | 8500.00 |

**其他支持格式**：CSV 文本、JSON、纯文本描述

**采样数据要求**：
- 覆盖各种状态值（如 status 的 1、2、3 都要有）
- 主键、外键、状态字段、金额字段优先提供
- 敏感数据需脱敏（证件号、手机号等）

**完整示例**：参见 [examples/sample_data.md](examples/sample_data.md)

## 路由逻辑

根据用户输入内容，自动判断执行哪些步骤：

| 用户输入情况 | 执行步骤 | 说明 |
|-------------|---------|------|
| DDL（首次建模） | Step 1 → Step 2 → Step 3 | 全量执行 |
| DDL + 采样数据 | Step 1 → Step 2 → Step 3 | 全量执行，置信度更高 |
| 已有业务过程清单 | Step 2 → Step 3 | 跳过 Step 1，用户提供的清单直接作为 Step 2 输入 |
| 已有数据域划分 | Step 3 | 跳过 Step 1、2，用户提供的划分直接作为 Step 3 输入 |
| 用户指定"只做业务过程识别" | Step 1 | 仅执行指定步骤 |
| 用户指定"只做数据域划分" | Step 2 | 仅执行指定步骤 |
| 用户指定"只做总线矩阵" | Step 3 | 仅执行指定步骤 |

**路由判断方法：**

- **Step A: 内容推断** — 根据用户提供的文件/文本类型判断：
  - 提供了 DDL → 从 Step 1 开始
  - 提供了业务过程清单（含"业务过程"、"事件"等关键词的表格） → 从 Step 2 开始，**该清单作为 Step 2 的输入**
  - 提供了数据域划分表（含"数据域"关键词的表格） → 从 Step 3 开始，**该划分作为 Step 3 的输入**

- **Step B: 用户排除** — 用户明确指定范围时，按用户要求执行

- **Step C: 默认全量** — 无法判断时，默认从 Step 1 开始全量执行

### 执行流程

**Step 1: 确认输入与范围**

1. 确认用户提供了哪些输入
2. 根据路由逻辑确定执行范围
3. 告知用户预计执行哪些步骤

**执行步骤**

按顺序执行各步骤，每个步骤完成后：
- 输出该步骤的产出文件
- 汇总置信度分布和 `❓` 标记数量
- 询问用户是否确认当前结果、是否需要调整
- 用户确认后进入下一步

**汇总输出**

所有步骤执行完毕后：
- Step 3 的产出文档中包含建模总结，汇总所有产出和待确认问题
- 整理待人工确认问题清单（按优先级排序）
- 给出后续步骤建议

## 注意事项

1. **不假设**：仅基于用户提供的 DDL 和采样数据推断，不添加不存在的表或字段
2. **标依据**：每个推断都标注依据来源
3. **多标 ❓**：宁可多标记不确定项，也不给错误结论
4. **分批处理**：表数量 >30 张时，按业务板块分批处理，每批确认后再继续
5. **参考不照搬**：行业模板仅作参考，需根据用户实际数据调整
