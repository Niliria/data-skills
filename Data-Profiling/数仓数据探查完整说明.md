## 一、业务背景与数据库概览

### 1.1 业务流程全景

```
投保申请 → 核保审核 → 承保出单 → 保费收取 → 保单生效
                ↓                                    ↓
            拒保终止              ┌──────────────────┤
                                 │        │          │
                              批改      出险       续保
                                 │        │          │
                                 │   报案→调查→定损→赔付
                                 │                   │
                          保单到期 / 退保 / 失效
```

### 1.2 源系统数据库概览

经过初步扫描，源系统共包含 **6 个业务数据库**，**68 张业务表**：

| 数据库名 | 业务含义 | 表数量 | 核心实体 |
|---------|---------|--------|---------|
| `db_policy` | 保单核心系统 | 18张 | 保单、被保人、险种 |
| `db_claim` | 理赔系统 | 14张 | 理赔案件、赔付记录 |
| `db_customer` | 客户系统 | 8张 | 客户、联系方式 |
| `db_agent` | 代理人系统 | 10张 | 代理人、佣金、考核 |
| `db_finance` | 财务系统 | 12张 | 收付款、账单 |
| `db_config` | 配置系统 | 6张 | 产品、费率、机构 |

## 二、模拟表结构（DDL）

### 2.1 投保申请表 `t_policy_apply`

```sql
CREATE TABLE db_policy.t_policy_apply (
    apply_id        VARCHAR(32)    NOT NULL COMMENT '申请单号（主键）',
    policy_no       VARCHAR(20)             COMMENT '保单号（核保通过后回填）',
    cust_id         VARCHAR(20)    NOT NULL COMMENT '客户ID',
    product_code    VARCHAR(10)    NOT NULL COMMENT '产品编码',
    agent_code      VARCHAR(15)    NOT NULL COMMENT '代理人编码',
    org_code        VARCHAR(10)    NOT NULL COMMENT '出单机构编码',
    channel_code    VARCHAR(8)              COMMENT '渠道编码',
    apply_amt       DECIMAL(14,2)  NOT NULL COMMENT '申请保额（元）',
    prem_amt        DECIMAL(12,2)  NOT NULL COMMENT '申请保费（元）',
    pay_mode        VARCHAR(4)              COMMENT '缴费方式：01年缴 02半年缴 03月缴',
    apply_date      DATE           NOT NULL COMMENT '申请日期',
    apply_status    VARCHAR(4)     NOT NULL COMMENT '状态：01待核保 02核保中 03承保 04拒保 05撤件',
    create_time     DATETIME       NOT NULL COMMENT '创建时间',
    update_time     DATETIME                COMMENT '最后更新时间',
    remark          VARCHAR(500)            COMMENT '备注',
    PRIMARY KEY (apply_id)
) COMMENT '投保申请表';
```

### 2.2 保单主表 `t_policy_main`

```sql
CREATE TABLE db_policy.t_policy_main (
    policy_no        VARCHAR(20)    NOT NULL COMMENT '保单号（主键）',
    apply_id         VARCHAR(32)    NOT NULL COMMENT '来源申请单号',
    cust_id          VARCHAR(20)    NOT NULL COMMENT '投保人客户ID',
    product_code     VARCHAR(10)    NOT NULL COMMENT '产品编码',
    agent_code       VARCHAR(15)    NOT NULL COMMENT '代理人编码',
    org_code         VARCHAR(10)    NOT NULL COMMENT '承保机构编码',
    channel_code     VARCHAR(8)              COMMENT '渠道编码',
    policy_status    VARCHAR(4)     NOT NULL COMMENT '保单状态：01有效 02失效 03退保 04到期 05批改中',
    sum_insured      DECIMAL(14,2)  NOT NULL COMMENT '保险金额（元）',
    prem_amt         DECIMAL(12,2)  NOT NULL COMMENT '保费（元）',
    std_prem         DECIMAL(12,2)           COMMENT '标准保费（元）',
    pay_mode         VARCHAR(4)              COMMENT '缴费方式',
    pay_years        SMALLINT                COMMENT '缴费年限',
    insure_years     SMALLINT                COMMENT '保险年限',
    effect_date      DATE           NOT NULL COMMENT '生效日期',
    expire_date      DATE           NOT NULL COMMENT '到期日期',
    underwrite_date  DATE                    COMMENT '承保日期',
    issue_date       DATE                    COMMENT '出单日期',
    create_time      DATETIME       NOT NULL COMMENT '创建时间',
    update_time      DATETIME                COMMENT '最后更新时间',
    PRIMARY KEY (policy_no),
    INDEX idx_cust   (cust_id),
    INDEX idx_agent  (agent_code),
    INDEX idx_effect (effect_date)
) COMMENT '保单主表';
```

### 4.9 数据时效性探查 → 确定调度时序

**探查操作：**

```sql
-- 评估业务发生到数据落库的延迟
SELECT
    AVG(DATEDIFF(create_time, effect_date))              AS avg_lag_days,
    MAX(DATEDIFF(create_time, effect_date))              AS max_lag_days,
    PERCENTILE(DATEDIFF(create_time,effect_date), 0.99) AS p99_lag,
    SUM(CASE WHEN DATEDIFF(create_time,effect_date)>1 THEN 1 ELSE 0 END)*100.0/COUNT(*) AS t2_pct
FROM db_policy.t_policy_main;

-- 各分区数据量稳定性
SELECT dt, COUNT(*) AS row_cnt,
       LAG(COUNT(*)) OVER(ORDER BY dt) AS prev_cnt,
       ROUND((COUNT(*)-LAG(COUNT(*)) OVER(ORDER BY dt))*100.0/LAG(COUNT(*)) OVER(ORDER BY dt),2) AS wow_pct
FROM db_policy.t_policy_main
WHERE effect_date >= '2024-06-01'
GROUP BY DATE(effect_date);
```

**探查结果：**

```
avg_lag_days  max_lag_days  p99_lag  t2_pct（T+2才到）
    1.3天          8天         3天      7.8%
```

**✅ 探查发现 9：调度时序建议**

> 99%的数据在 T+3 天内完全到达，7.8%的数据在 T+2 才落库。
> **建议：** DWS 层调度时间设置为 **T+2 的 06:00**，
> 不能设置为 T+1，否则每天有近 8% 的保单数据会缺失，

### 6.3 DWM 层整合设计（核心宽表）

#### 保单生命周期表 `dwm_policy_lifecycle_df`

整合来源：`dwd_policy_main + dwd_policy_apply + dwd_policy_status_log + dwd_premium_recv`

关键字段设计：

```sql
CREATE TABLE dwm_policy_lifecycle_df (
    policy_no          STRING    COMMENT '保单号',
    apply_id           STRING    COMMENT '申请单号',
    -- 投保信息
    cust_id            STRING    COMMENT '投保人ID',
    product_code       STRING    COMMENT '产品编码',
    agent_code         STRING    COMMENT '代理人编码（承保时）',
    org_code           STRING    COMMENT '承保机构编码',
    channel_code       STRING    COMMENT '渠道编码',
    -- 时间节点（全部保留）
    apply_date         STRING    COMMENT '申请日期',
    underwrite_date    STRING    COMMENT '承保日期',
    effect_date        STRING    COMMENT '生效日期',
    expire_date        STRING    COMMENT '到期日期',
    -- 当前状态
    policy_status      STRING    COMMENT '当前状态',
    is_valid           TINYINT   COMMENT '是否有效件：1是 0否',
    is_surrendered     TINYINT   COMMENT '是否退保：1是 0否',
    is_lapsed          TINYINT   COMMENT '是否失效：1是 0否',
    is_revived         TINYINT   COMMENT '是否曾复效：1是 0否',  -- 来自status_log探查
    -- 金额信息
    sum_insured        DECIMAL   COMMENT '保险金额',
    prem_amt           DECIMAL   COMMENT '保费',
    std_prem           DECIMAL   COMMENT '标准保费',
    -- 收费信息（聚合自 dwd_premium_recv）
    total_recv_prem    DECIMAL   COMMENT '累计实收保费',
    recv_years         INT       COMMENT '已缴年度数',
    last_recv_date     STRING    COMMENT '最近一次缴费日期',
    -- 理赔信息（聚合自 dwd_claim）
    claim_cnt          INT       COMMENT '出险次数',
    total_claim_amt    DECIMAL   COMMENT '累计赔付金额',
    is_claimed         TINYINT   COMMENT '是否曾出险：1是 0否',
    -- 数据管理
    dt                 STRING    COMMENT '数据日期'

COMMENT 'DWM-保单生命周期宽表'
PARTITIONED BY (dt STRING);
```

## 七、访谈提纲（探查自动生成）

基于以上探查，以下问题必须在开发前通过业务访谈确认：

### 🔴 必须确认（高优先级）

**Q1. policy_status = 'XB' 是什么含义？**
数据现象：125条保单状态为 'XB'，DDL注释中未定义此枚举值。
需确认：该状态是已退市的历史状态？还是在用的特殊业务状态（如信用卡保险批次）？
影响决策：是否计入有效件数、是否过滤。

**Q2. channel_code 空值 3.2% 的原因是什么？**
数据现象：4,000条保单无渠道编码，且集中在2019-2020年。
需确认：是旧系统未记录渠道，还是存在一种无渠道的直销模式？
影响决策：渠道维度缺失会导致渠道分析报表不全。

**Q3. 7.8% 的数据 T+2 才到达，是否可以接受 T+2 的报表时效？**
数据现象：平均数据延迟1.3天，7.8%的保单在T+2日才落入系统。
需确认：业务方能否接受日报数据为前天数据，还是必须要T+1？
影响决策：若必须T+1，需要与源系统团队沟通优化数据链路。

**Q4. agent_code 空值的1,065条是什么业务场景？**
数据现象：1,065条保单无代理人编码，占比0.85%。
需确认：是网销直投（无代理人）？还是公司直销？还是数据录入遗漏？
影响决策：代理人业绩指标是否将这部分计入特定渠道。

### 🟡 建议确认（中优先级）

**Q5. prem_amt = 0 的10条保单是否为团险豁免保费？**
数据现象：10条保单保费为0，保额正常。
推断：可能是团体险主险豁免附加险保费的场景。
影响决策：新单保费指标是否将这10条过滤。

**Q6. 新单件数的时间口径是承保日还是生效日？**
业务背景：两个口径数值差异在0.5%-2%之间。
需确认：管理层驾驶舱、监管报表、内部考核各用什么口径。
影响决策：DWS层需要同时保留两个时间字段的聚合结果。

**Q7. 历史12条理赔孤儿记录是否需要补录对应保单？**
数据现象：12条2019年理赔记录找不到对应保单（旧系统迁移遗留）。
需确认：这12条对应的历史保单是否有旧系统数据可以补录。
影响决策：影响2019年理赔率的历史数据准确性。
